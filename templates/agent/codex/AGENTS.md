# プロジェクトルール

## コーディングルール

### 既存規約への追従

プロジェクト内の既存のコードスタイル、命名規則、パターンに従うこと。

### 変更スコープの制限

現在のタスクに無関係なファイルを編集してはならない。

### コメント

コードの内容をそのまま繰り返すだけの自明・冗長なコメントを追加してはならない。

## セキュリティルール

### .envファイル

`.env`ファイルおよびその内容を読み込み・表示・参照してはならない。
これらのファイルにはシークレットや機密情報が含まれている可能性がある。

### シークレット・認証情報

APIキー、トークン、パスワード等の秘密情報をコードやコミットメッセージに含めてはならない。
`.npmrc`、`credentials.json`等の認証情報ファイルも同様に読み込み・参照してはならない。

## Gitルール

### Push

ユーザーから明示的な指示がない限り、GitHubへpushしてはならない。
mainブランチへ直接pushする場合は、事前にユーザーへ警告すること。

### 破壊的操作

`--force` push、`--hard` reset等の破壊的・不可逆なgitコマンドは、ユーザーから明示的に要求されない限り使用してはならない。

### コミットメッセージ

コミットメッセージは英語で記述すること。
内容を要約し、最長でも3行程度に収めること。変更のすべてを網羅する必要はない。

## コマンド実行ルール

### ファイルシステムの破壊的操作

`rm -rf` によるディレクトリの再帰削除、および大量ファイルの一括削除は、ユーザーから明示的に指示されない限り実行してはならない。
プロジェクト外のパスに対する削除操作は、明示的な指示があっても実行前にユーザーへ確認すること。

### 特権コマンド

`sudo` および `su` による権限昇格を伴うコマンドは、ユーザーから明示的に指示されない限り実行してはならない。

### パーミッション変更

`chmod 777` など全ユーザーに書き込み権限を付与する操作、およびシステムパス（`/etc`、`/usr`、`/var` 等）に対する `chown` は実行してはならない。

### グローバルパッケージのインストール

`npm install -g` によるグローバルインストール、および仮想環境（virtualenv / venv）が有効でない状態での `pip install` は、ユーザーから明示的に指示されない限り実行してはならない。

### プロセスの強制終了

`kill -9` でプロジェクト外のプロセスを強制終了してはならない。

### システム操作

`shutdown`、`reboot`、`systemctl stop` 等のシステム全体に影響するコマンドは実行してはならない。

## ワークフロー

### コミット（/commit）

ステージされた変更を確認し、conventional commits形式でコミットを作成する。

1. `git diff --staged` で変更内容を確認する
2. 変更の種別（feat / fix / refactor / docs / chore 等）を判断する
3. コミットメッセージを英語で1行（最長72文字）で作成する
4. `git commit -m "..."` を実行する

### PR説明文（/pr）

現在のブランチの変更をもとにPull Requestの説明文を作成する。

1. `git log main..HEAD --oneline` でコミット一覧を確認する
2. `git diff main...HEAD` で差分全体を把握する
3. 以下の形式でPR説明文を出力する:

```
## Summary
- （変更の概要を箇条書き）

## Changes
- （具体的な変更点を箇条書き）

## Test plan
- （確認すべきことを箇条書き）
```
